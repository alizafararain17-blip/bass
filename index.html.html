Platform manifest

Everything on the ship, nothing on the shelf

No add-on marketplace to dig through. The tools you'd reach for on day one are already in the base image.

CNT-01 / COMPUTE

Container-native runtime

Any Dockerfile ships as-is. No buildpack guessing games, no framework allowlist.

CNT-02 / NETWORK

Global anycast edge

Requests land at the nearest of 20+ regions automatically, with zero config routing.

CNT-03 / DATA

Managed Postgres & Redis

Provisioned alongside your app, backed up hourly, reachable over a private network.

CNT-04 / CI

Preview per pull request

Every PR gets a disposable environment with its own URL, torn down on merge.

CNT-05 / OBSERVE

Structured logs & traces

Query logs like a database. Pipe them to your own sink with one line of config.

CNT-06 / CONTROL

Full CLI and API

Everything the dashboard can do, the CLI can do — scriptable, so your pipeline owns it.

Shipping route

From commit to live, three stops

No staging tickets, no manual promotion steps. The route is the same for a solo project and a 40-service backend.

STOP 01

Connect the repo

Point Dockyard at a GitHub, GitLab, or bare git remote. It reads your Dockerfile or dockyard.toml — no rewrite required.

dockyard link ./api-service
STOP 02

Push a branch

Every push builds. main goes to production; every other branch gets its own preview URL automatically.

git push dockyard main
STOP 03

Watch it land

Follow the rollout in real time, region by region, with automatic rollback if health checks fail.

dockyard logs --follow
